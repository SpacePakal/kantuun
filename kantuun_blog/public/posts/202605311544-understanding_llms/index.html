1. The Two Lives of a Model: Training vs. Running

1.1 Training: Teaching the Model

1.1.1 What Training Actually Does

  • Billions of forward and backward passes
  • Weights are updated iteratively via backpropagation
  • The model’s intelligence, knowledge, and reasoning ceiling are set here — permanently

1.1.2 Why Training is Compute-Bound

  • Requires massive parallelism: gradient calculations, weight updates, activation storage
  • The bottleneck is raw computational throughput (TFLOPS)
  • VRAM must hold: weights + gradients + activations + optimizer states (4–6x model size)

1.1.3 Why Training Needs a Dedicated GPU

  • The entire ecosystem (PyTorch, JAX, LoRA, QLoRA) is built around NVIDIA CUDA
  • Apple Silicon (MLX) and integrated GPUs work but are significantly slower
  • An older CUDA GPU (e.g. RTX 2070 Super) outperforms modern integrated GPUs for training

1.2 Inference: Running the Model

1.2.1 What Inference Actually Does

  • Weights are frozen — only a forward pass happens
  • Each token generated requires reading the full model weights from memory
  • The bottleneck is how fast you can move data, not how fast you can compute

1.2.2 Why Inference is Memory-Bandwidth-Bound

  • Speed depends on how quickly weights stream into compute units
  • Unified memory architectures (Apple Silicon, Framework) have high bandwidth to integrated GPU
  • This is why they punch above their weight for inference despite weaker compute

1.2.3 The KV Cache: Where Your Context Lives

  • Every token in the active conversation is stored in RAM as a key-value cache
  • Longer context = more RAM consumed at runtime
  • This is why a model trained with 128K context may only handle 8K in practice on limited RAM

2. Memory: The Runtime Dimension

2.1 What RAM Controls at Inference Time

2.1.1 Model Fit

  • Model weights must fully reside in memory
  • A 70B model at Q4 quantization ≈ 40GB — won’t fit in a 24GB GPU VRAM card
  • Unified memory (64GB, 128GB) can hold models that discrete GPUs cannot

2.1.2 Context Length

  • More RAM = longer conversations and larger documents you can feed the model
  • Context is the only dimension of “capability” you can expand at runtime

2.1.3 Quantization Quality

  • More RAM lets you run higher-precision quantization (Q8 vs Q4)
  • Q8 is closer to original trained weights → less quality loss
  • This is the closest thing to “more RAM = better output” — but you’re losing less, not gaining

2.2 What RAM Cannot Do

  • RAM cannot make a model smarter
  • RAM cannot expand the model’s knowledge beyond its training cutoff
  • RAM cannot improve reasoning depth — that ceiling was set at training

3. Model Size and Precision: Weights as Data

3.1 How Model Size Maps to Parameters

  • Each parameter stored in fp16 = 2 bytes
  • A 7B parameter model in fp16 ≈ 14GB
  • A 70B parameter model in fp16 ≈ 140GB

3.2 Quantization: Trading Precision for Fit

3.2.1 What Quantization Does

  • Reduces bytes-per-parameter (fp16 → int8 → int4)
  • Shrinks model footprint at the cost of some fidelity to original weights

3.2.2 Quantization Levels in Practice

Format Bytes/Param 70B Model Size Quality
fp16 2.0 ~140GB Full
Q8 1.0 ~70GB Near-full
Q4 0.5 ~40GB Good
Q2 0.25 ~20GB Degraded

4. Hardware Archetypes and Their LLM Role

4.1 Discrete NVIDIA GPU

  • Best for: training, fine-tuning, LoRA/QLoRA
  • Constraint: fixed VRAM (24GB on RTX 4090) — hard ceiling for model size
  • Strength: CUDA ecosystem, raw TFLOPS

4.2 Unified Memory (Apple Silicon, Framework Max+)

  • Best for: inference, portable local LLMs
  • Strength: RAM doubles as VRAM — 64GB or 128GB accessible to the GPU
  • Constraint: slower compute, limited training ecosystem (MLX only)

4.3 Cloud GPU (RunPod, Lambda Labs, Vast.ai)

  • Best for: serious fine-tuning without owning the hardware
  • Renting A100/H100 time is often more practical than buying for occasional training runs

5. The Intelligence Ceiling: Fixed at Training

5.1 What Training Defines Forever

  • Knowledge cutoff
  • Reasoning depth and style
  • Language and task capabilities

5.2 What Runtime Can Tune

  • Context length (more RAM)
  • Output precision (higher quantization)
  • Model choice (bigger model = better reasoning, requires more RAM)

5.3 The Core Mental Model

Training sets the ceiling. Runtime determines how close you get to it.

6. Learning Resources for First Principles

6.1 Bottom-Up: Build a GPT from Scratch

  • Andrej Karpathy — Neural Networks: Zero to Hero (YouTube)
  • Builds a transformer from scratch in PyTorch
  • Hardware intuitions emerge naturally from watching the code run

6.2 Top-Down: Train Real Models First, Understand Later

  • Deep Learning for Coders with fastai and PyTorch — Jeremy Howard & Sylvain Gugger
  • Practical-first, minimal math prerequisites
  • Covers vision, NLP, tabular — broad foundation
  • Companion course at course.fast.ai is kept more current than the book

6.3 How They Complement Each Other

  • fastai/Howard: breadth across domains, intuition through application
  • Karpathy: depth on transformers and language models specifically
  • Together: a complete mental model from use case down to matrix multiplication
